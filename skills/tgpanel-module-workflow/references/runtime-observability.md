# 后台 Worker 可观测性

常驻监听、轮询、采集、同步和通知模块必须同时证明“循环活着”和“业务确实执行”。
仅显示最近轮询时间不足以判断模块是否可用。

## 标准运行态

管理 API 至少提供以下字段，名称可按模块语言风格调整，但语义必须保持：

- `Status`、`StatusMessage`：`starting`、`disabled`、`idle`、`running`、
  `success`、`warning`、`error` 或 `backoff`。
- `LastHeartbeatAtUtc`：正在执行时持续推进，用于发现卡死。
- `LastPollAtUtc`：后台循环最近一次被调度并检查配置的时间。
- `LastRunStartedAtUtc`、`LastRunFinishedAtUtc`：一轮完整业务的边界。
- `LastBusinessChangeAtUtc`：最近一次真正写入、转发、邀请或通知成功的时间。
- `LastError`：最近原始根因，不使用“退避中”等状态文字覆盖。
- `ConsecutiveErrors`、`NextRetryAtUtc`：连续失败次数与下次重试时间。
- `CurrentStage`：当前任务、账号、来源或处理阶段；不得包含凭据。
- `LastSummary`：上一轮输入、处理、跳过、成功、失败和业务变化计数。

`LastPollAtUtc`、`LastRunFinishedAtUtc` 与 `LastBusinessChangeAtUtc` 必须分开，
避免“循环还在跑”被误解为“刚刚采集到数据”。

## 状态更新规则

1. 轮询开始时更新 `LastPollAtUtc`；真正开始业务时再更新
   `LastRunStartedAtUtc`。
2. 网络等待或长循环期间定期更新心跳和当前阶段。
3. 一轮结束时原子写入结束时间、Summary 和最终状态。
4. 没有新数据但完整检查成功属于 `success`，Summary 必须能证明检查范围。
5. 部分来源无法解析、被跳过或部分失败属于 `warning` 或 `error`，不能伪报成功。
6. 退避期间保留上一次原始异常，同时展示 `NextRetryAtUtc`；只有后续成功或
   用户明确重置时才清除根因和连续错误计数。
7. 在生成 Summary 前抛出的异常也要形成失败 Summary，不能继续展示旧的全零结果。
8. 进程重启后不要恢复 `running`；恢复累计统计、游标和上一轮结果，再从
   `starting/idle` 重新进入状态机。

## 手动诊断入口

提供受管理员鉴权保护的“立即运行一次”接口，并遵守：

- 后台循环和手动触发共享同一个互斥锁，禁止重叠执行。
- 先取得互斥锁，再修改 `Status`、`LastError` 或当前轮 Summary；锁获取失败时只返回明确的忙碌结果，不得覆盖正在执行任务的状态。
- 支持取消和超时；接口不得无限等待。
- 返回本次完整 Summary，页面随后刷新运行态。
- 诊断接口默认执行真实业务；若支持 dry-run，必须明确标识且不得混淆结果。

## 游标与幂等

- 为每个来源分别持久化游标，不用一个全局时间代表全部进度。
- 只在业务提交成功后推进游标；失败时保留可重试位置。
- 对用户名、消息、成员或通知键执行幂等去重。
- 对缓存、错误明细和历史记录设置上限，避免常驻服务内存无限增长。

## 最低测试集

- 禁用、无配置和空数据。
- 成功但无业务变化。
- 部分跳过/部分失败。
- 在 Summary 创建前抛出异常。
- 连续失败进入退避且仍保留原始错误。
- 手动触发与后台循环并发。
- 取消、宿主重启和游标恢复。
- 心跳停止时页面能提示状态过期。

生产验收不能只要求 HTTP 200。至少完成一轮真实业务，并根据模块配置断言
检查数、跳过数、错误数和业务结果。
